# Project 1 Requirements Coverage Matrix

This matrix maps common AI Automation Engineer role requirements to concrete
evidence in Project 1.

| Requirement / Skill | Evidence in Project 1 | Files / Modules | Demo Talking Point |
| --- | --- | --- | --- |
| Python automation | Python 3.12+ project with typed modules, orchestration, utilities and local scripts. | `01-ai-marketing-ops-agent/src/marketing_ops_agent/`, `pyproject.toml`, `scripts/` | The workflow is an executable Python automation system, not a notebook or chatbot wrapper. |
| Playwright browser automation | Async scraper logs into the local HTML-only marketing panel and extracts campaign rows. | `browser/panel_scraper.py`, `tests/browser/test_panel_scraper.py`, `mock_services/marketing_panel.py` | Playwright is used only where no API exists: the local mock marketing panel. |
| REST API integration | Typed `httpx` clients call Campaign and Project Management REST APIs with validation and error handling. | `clients/campaign_client.py`, `clients/project_management_client.py`, `mock_services/campaign_api.py`, `mock_services/project_management_api.py` | API integration is preferred over browser automation when an API is available. |
| GraphQL integration | Typed Analytics GraphQL client fetches campaign metrics from the local mock GraphQL endpoint. | `clients/analytics_client.py`, `mock_services/analytics_graphql_api.py`, `tests/clients/test_analytics_client.py` | The workflow combines REST and GraphQL data sources into one validated business object. |
| Workflow orchestration | Daily workflow coordinates scraping, clients, aggregation, anomaly detection, reporting, approvals, optional LLM and notifications. | `workflows/daily_marketing_report.py`, `workflows/models.py`, `tests/workflows/test_daily_marketing_report.py` | The orchestration layer uses dependency injection so tests can replace external boundaries with fakes. |
| Deterministic validation | Source data is aggregated into `CampaignSnapshot` objects with data quality flags and human-review signals. | `aggregation/aggregator.py`, `aggregation/campaign_snapshot.py`, `aggregation/data_quality.py`, `tests/aggregation/` | Missing or inconsistent data is flagged explicitly instead of silently dropped or inferred. |
| Anomaly detection | Rule-based detector emits typed `AnomalyFinding` objects for performance and data quality issues. | `anomaly/detector.py`, `anomaly/rules.py`, `anomaly/models.py`, `tests/anomaly/` | CPA, ROI, high-spend and data quality findings are deterministic and testable. |
| Report generation | Deterministic Markdown report renders executive summary, anomalies, grouped data quality issues, human review, snapshots and limitations. | `reporting/markdown_report.py`, `reporting/models.py`, `tests/reporting/test_markdown_report.py` | The report is generated from validated snapshots and findings, not from raw payloads or LLM guesses. |
| Optional LLM interpretation | LLM layer interprets validated outputs only and uses a deterministic mock provider for local/test runs. | `llm/interpreter.py`, `llm/providers.py`, `llm/prompts.py`, `tests/llm/` | LLM interpretation is optional, downstream and not the source of truth for metrics or findings. |
| Human approval / guardrails | Approval service creates pending requests for critical findings, human-review findings and high-risk LLM recommendations. | `approval/models.py`, `approval/service.py`, `approval/approval_store.py`, `tests/approval/` | Pending approval requests are explicit local records and are not treated as approved actions. |
| Approval-aware notification delivery | Optional notification service sends summary-only mock notifications with pending approval IDs when present. | `notifications/models.py`, `notifications/providers.py`, `notifications/service.py`, `tests/notifications/` | Real Slack, Telegram and email providers are not used; the provider is deterministic and mocked. |
| Observability / run history | Workflow records structured JSONL run history with status, counts, report path, approval count and sanitized failures. | `observability/models.py`, `observability/run_recorder.py`, `tests/observability/`, `run-history/.gitkeep` | Every local run can be audited through `run-history/workflow-runs.jsonl`. |
| CI/CD | GitHub Actions validates Project 1 on relevant pull requests and pushes to `main`. | `.github/workflows/project-1-ci.yml`, `scripts/run_ci_locally.sh` | CI runs pytest, ruff, mypy, Playwright Chromium install, Docker Compose config and shell syntax checks. |
| Docker | Docker Compose runs local FastAPI mock services for the marketing panel, REST APIs and GraphQL API. | `compose.yaml`, `Dockerfile.mock`, `mock_services/` | The demo environment is reproducible without real third-party services. |
| Testing | Test suite covers utilities, models, services, clients, scraper fakes, aggregation, anomaly detection, reporting, workflow, approvals, LLM and notifications. | `tests/` | The project is built around deterministic tests rather than manual-only demos. |
| Documentation | Root docs, project README, architecture, decisions, runbook, case study, demo script and coverage matrix document the design and demo flow. | `README.md`, `docs/`, `01-ai-marketing-ops-agent/docs/` | Reviewers can inspect both the implementation and the reasoning behind it. |
| Safe handling of secrets | No real secrets are hardcoded. `.env.example` uses mock local defaults and empty placeholders. Failure and approval text are sanitized before persistence. | `.env.example`, `config.py`, `observability/models.py`, `approval/models.py`, `.gitignore` | The local demo uses mock credentials only, and generated runtime files are kept out of git. |
